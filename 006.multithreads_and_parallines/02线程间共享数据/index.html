2线程间共享数据

  1. 共享数据带来的问题
  2. C++ 标准中定义了数据竞争这个术语，一种特殊的条件竞争：并发的去修改一个独立对象，数据竞争是(可怕的)未定义行为的起因
  3. 最简单的办法就是对数据结构采用某种保护机制，确保只有进行修改的线程才能看到不变量被破坏时的中间状态。从其他访问线程的角度来看，修改不是已经完成了，就是还没开始。另一个选择是对数据结构和不变量的设计进行修改，修改完的结构必须能完成一系列不可分割的变化，也就是保证每个不变量保持稳定的状态，这就是所谓的无锁编程。不过，这种方式很难得到正确的结果。
  4. 另一种处理条件竞争的方式是，使用事务的方式去处理数据结构的更新(这里的”处理”就如同对数据库进行更新一样)。所需的一些数据和读取都存储在事务日志中，然后将之前的操作合为一步，再进行提交。当数据结构被另一个线程修改后，或处理已经重启的情况下，提交就会无法进行，这称作为“软件事务内存”(software transactionalmemory (STM))。理论研究中，这是一个很热门的研究领域。这个概念将不会在本书中再进行介绍，因为在 C++ 中没有对STM进行直接支持.
  5. 保护共享数据结构的最基本的方式，是使用C++标准库提供的互斥量。
  6. 使用互斥量保护数据(锁相关内容重点)
  7. 1使用互斥量：std::mutex(不可拷贝/移动) 成员函数lock() unlock()但是不建议直接用，raii的模版lock_gard unique_gard shared_gard scoped_gard
    • 在大多数情况下，互斥量通常会与需要保护的数据放在同一类中，而不是定义成全局变量。这是面向对象设计的准则：将其放在一个类中，就可让他们联系在一起，也可对类的功能进行封装，并进行数据保护。互斥量和需要保护的数据，在类中都定义为private成员，这会让访问数据的代码更清晰，并且容易看出在什么时候对互斥量上锁。当所有成员函数都会在调用时对数据上锁，结束时对数据解锁，这就保证了访问时数据不变量不被破坏。事情也不是总是那么理想，聪明的你一定注意到了：当其中一个成员函数返回的是保护数据的指针或引用时，会破坏数据。具有访问能力的指针或引用可以访问(并可能修改)被保护的数据，而不会被互斥锁限制。这就需要对接口有相当谨慎的设计，要确保互斥量能锁住数据的访问，并且不留后门。
  8. 2构建代码以保护数据。
    • 检查指针和引用。只要没有成员函数通过返回值或者输出参数的形式，向其调用者返回指向受保护数据的指针或引用，数据就是安全的。如果你还想深究，就没这么简单了。确保成员函数不会传出指针或引用的同时，检查成员函数是否通过指针或引用的方式来调用也是很重要的(尤其是这个操作不在你的控制下时)。函数可能在没互斥量保护的区域内，存储着指针或者引用，这样就很危险。更危险的是：将保护数据作为一个运行时参数。从乐观的角度上看，还是有方法可循的：切勿将受保护数据的指针或引用传递到互斥锁作用域之外，无论是函数返回值，还是存储在外部可见内存，亦或是以参数的形式传递到用户提供的函数中去。
  9. 3定位接口间的条件竞争
    • 例如，构建一个类似于 std::stack 结构的栈，除了构造函数和swap()以外，需要对 std::stack 提供五个操作：push()一个新元素进栈，pop()一个元素出栈，top()查看栈顶元素，empty()判断栈是否是空栈，size()了解栈中有多少个元素。即使修改了top()，使其返回一个拷贝而非引用(2的准则)，对内部数据使用一个互斥量进行保护，不过这个接口仍存在条件竞争。这个问题不仅存在于基于互斥量实现的接口中，在无锁实现的接口中，条件竞争依旧会产生。这是接口的问题，与其实现方式无关。对于共享的栈对象，因为在调用empty()和调用top()之间，可能有来自另一个线程的pop()调用删除了最后一个元素。这是一个经典的条件竞争，使用互斥量对栈内部数据进行保护，但依旧不能阻止条件竞争的发生，这就是接口固有的问题。当仔细的观察过之前的代码段，就会发现另一个潜在的条件竞争在调用top()和pop()之间。假设两个线程运行着前面的代码，并且都引用同一个栈对象s。...太冗长了去看原文，总之就是锁的粒度太小导致了条件竞争。细粒度锁定方案的一个问题是，有时为了保护操作中的所有数据，需要锁定多个互斥锁。如前所述，有时正确的做法是增加互斥锁保护的数据的粒度，以便只需锁定一个互斥锁。然而，有时这是不可取的，例如当互斥锁保护的是类的不同实例时。在这种情况下，在更高的级别上锁定意味着要么将锁定留给用户，要么有一个保护该类所有实例的单个互斥锁，这两者都不是特别理想。
  10. 4死锁：问题及解决方法
    • 一对线程需要对他们所有的互斥量做一些操作，其中每个线程都有一个互斥量，且等待另一个解锁。这样没有线程能工作，因为他们都在等待对方释放互斥量。这种情况就是死锁，它的最大问题就是由两个或两个以上的互斥量来锁定一个操作。
    • 避免死锁的一般建议，就是让两个互斥量总以相同的顺序上锁：总在互斥量B之前锁住互斥量A，就永远不会死锁。
    • C++标准库有办法解决这个问题， std::lock ——可以一次性锁住多个(两个以上)的互斥量，并且没有副作用(死锁风险)。尽管 std::lock 可以帮助您避免在需要一起获取两个或更多锁的情况下发生死锁，但如果它们是单独获取的，它就无能为力。在这种情况下，您必须依靠作为开发人员的自律，以确保您不会遇到死锁。这并不是一件容易的事，然而，确实有一些相对简单的规则可以帮助您编写无死锁代码。
  11. 5避免死锁进阶指导
    • 死锁通常是由于对锁的不当操作造成，但也不排除死锁出现在其他地方。无锁的情况下，仅需要每个 std::thread 对象调用join()，两个线程就能产生死锁。所以三个或更多线程的互相等待也会发生死锁。为了避免死锁，这里的指导意见为：当机会来临时，不要拱手让人。以下提供一些个人的指导建议，如何识别死锁，并消除其他线程的等待。
    • 1.避免嵌套锁：一个线程已获得一个锁时，再别去获取第二个。当你需要获取多个锁，使用一个std::lock来做这件事(对获取锁的操作上锁)，避免产生死锁。
    • 2.避免在持有锁时调用用户提供的代码：因为代码是用户提供的，你没有办法确定用户要做什么；用户程序可能做任何事情，包括获取锁。当你正在写一份通用代码，例如之前栈stack，每一个操作的参数类型，都在用户提供的代码中定义，就需要其他指导意见来帮助你。
    • 3.使用固定顺序获取锁：当硬性条件要求你获取两个或两个以上的锁，并且不能使用 std::lock 单独操作来获取它们；那么最好在每个线程上，用固定的顺序获取它们(锁)。例如链表中删除操作需要锁定三个相邻的节点，而遍历操作需要锁定当前节点和下一个节点，一旦下一个节点上的锁被获取，那么第一个节点的锁就可以释放了，因为没有持有它的必要性了。则它们必须按相同的顺序获取锁，否则会死锁。
    • 4.使用层级锁：虽然，定义锁的顺序是一种特殊情况，但层级锁的意义在于提供对运行时顺序约定的检查。你需要对应用进行分层，并识别在给定层上所有可上锁的互斥量。http://zhengyi.me/2018/04/11/what-is-lock-hierarchy/当代码尝试锁定一个互斥锁时，如果它已经持有来自更低层的锁，则不允许锁定该互斥锁。你可以通过为每个互斥锁分配层编号并记录每个线程锁定了哪些互斥锁，在运行时检查这一点。层次互斥锁之间的死锁因此是不可能的，因为互斥锁本身强制执行锁的顺序。这意味着如果两个锁处于层次的同一层级，你不能同时持有它们，因此手交手锁定方案要求链中的每个互斥锁拥有比前一个更低的层次值，这在某些情况下可能是不切实际的。一个用户定义的锁类型，它可以用于 std::lock_guard<> 模板中的要求是，它实现三个满足互斥量操作成员函数：lock(), unlock() 和 try_lock()。try_lock()：当互斥量上的锁被一个线程持有，它将返回false，而不是等待调用的线程，直到能够获取互斥量上的锁为止。在 std::lock() 的内部实现中，try_lock()会作为避免死锁算法的一部分。(原文中有层级锁的实现可以参考)。使用了thread_local的值来代表当前线程的层级值。它被初始化为最大值，所以最初所有线程都能被锁住。因为其声明中有thread_local，所以每个线程都有其拷贝副本，这样线程中变量状态完全独立，当从另一个线程进行读取时，变量的状态也完全独立。
    • 5将以上指南扩展到锁之外：死锁不仅仅发生在锁上；它可以发生在任何可能导致等待循环的同步结构中。因此，值得扩展这些指导原则以涵盖这些情况。例如，正如您应该尽量避免获取嵌套锁一样，持有锁时等待线程也是一个坏主意，因为该线程可能需要获取锁才能继续。同样，如果您要等待一个线程完成，识别线程层次结构可能是值得的，以便让一个线程只等待层次结构中的较低线程。实现这一点的一个简单方法是确保您的线程在启动它们的同一函数中被加入。一旦您设计了避免死锁的代码，std::lock()和std::lock_guard涵盖了大多数简单锁定的情况，但有时需要更大的灵活性。对于这些情况，标准库提供了std::unique_lock模板。它还提供了与std::lock_guard相同的RAII风格的锁管理，但提供了更多灵活性。
  12. 6用std::unique_lock实现灵活的锁定
    • 首先，可将 std::adopt_lock 作为第二个参数传入构造函数，对互斥量进行管理；也可以将std::defer_lock作为第二个参数传递进去，表明互斥量应保持解锁状态。这样，就可以被std::unique_lock对象(不是互斥量)的lock()函数所获取，或传递std::unique_lock对象到std::lock()中。
    • std::unique_lock实例不会总与互斥量的数据类型相关，实例中的标志确定该实例是否拥有特定的互斥量，这个标志是为了确保unlock()在析构函中被正确调用。如果实例拥有互斥量，那么析构函数必须调用unlock()；但当实例中没有互斥量时，析构函数就不会去调用unlock()。这个标志可以通过owns_lock()成员变量进行查询。除非你想将std::unique_lock的所有权进行转让，或是对其做一些其他的事情外，你最好使用C++17中提供的std::scoped_lock。
    • 当std::lock_guard已经能够满足你的需求时，还是建议你继续使用它。当需要更加灵活的锁时，最好选择std::unique_lock，因为它更适合于你的任务。你已经看到一个递延锁的例子，另外一种情况是锁的所有权需要从一个作用域转到另一个作用域。
    • std::unique_lock 的灵活性还允许实例在销毁之前放弃它们的锁。意味着如果该锁已经不再需要，您可以选择在特定的代码位置释放它。持有锁的时间超过需要的时间可能会导致性能下降，因为等待锁的其他线程会被阻短比必要的更长时间。
  13. 7不同作用域中互斥量所有权的传递
    • std::unique_lock 实例没有与自身相关的互斥量，一个互斥量的所有权可以通过移动操作，在不同的实例中进行传递。某些情况下，这种转移是自动发生的，例如：当函数返回一个实例；另些情况下，需要显式的调用std::move()来执行移动操作。
    • 一种使用可能是允许一个函数去锁住一个互斥量，并且将所有权移到调用者上，所以调用者可以在这个锁保护的范围内执行额外的动作。通常这种模式会在需要锁定的互斥量取决于程序当前状态或传递给返回 std::unique_lock 对象的函数的参数时使用。
    • 有这样的用法：锁并不是直接返回的，而是一个网关类的数据成员，用于确保对某些受保护数据的正确锁定访问。当你希望访问数据时，您需要获得一个网关类的实例。通过调用一个get_lock()函数，该实例将获取锁。然后，您可以通过网关对象的成员函数访问数据。访问完成后，销毁网关对象，这将释放锁并允许其他线程访问受保护的数据。这样的网关对象可能是可移动的（以便可以从函数返回），在这种情况下，锁对象(网关数据成员)也需要是可移动的(std::mutex不可拷贝移动，只能用unique_lock包装或者数据本身的移动,详细见stdmutex不支持移动语义.md)。
  14. 8以适当的粒度进行锁定
    • 在可能的情况下，只有在实际访问共享数据时才锁定互斥量；尽量在锁外进行数据处理。特别是，不要在持有锁的时候进行真正耗时的活动，例如文件输入/输出，或者另一个不需要同时锁定的锁。除非锁定确实是为了保护对文件的访问，在持有锁时执行输入/输出将不必要地延迟其他线程（因为它们将在等待获取锁时被阻塞），从而可能消除使用多线程的任何性能提升。std::unique_lock在这种情况下可以很好工作，代码不需要再访问共享数据时调用unlock()；当再次需要对共享数据进行访问时，就可以再调用lock()。
    • 相反，当你持有锁的时间没有达到整个操作时间，就会让自己处于条件竞争的状态。
    • 有时，适当粒度级别并不存在，因为并非所有对数据结构的访问都需要相同的保护级别。在这种情况下，可能更适合使用其他机制，而不是简单的 std::mutex。
  15. 保护共享数据的替代设施
  16. 1保护共享数据的初始化(但是后续不需要只读)
    • C++标准提供了一种纯粹保护共享数据初始化过程的机制。标准库提供了std::once_flag和std::call_once来处理这种情况。 std::call_once可以和任何函数或可调用对象一起使用。值得注意的是，std::mutex和std::once_flag的实例不能拷贝和移动，需要通过显式定义相应的成员函数，对类成员进行操作。
    • 很多在不支持C++11标准的编译器上,还有一种初始化过程中潜存着条件竞争：其中一个局部变量被声明为static类型，这种变量的在声明后就已经完成初始化；对于多线程调用的函数，这就意味着这里有条件竞争——抢着去定义这个变量。在C++11标准中，这些问题都被解决了：初始化及定义完全在一个线程中发生，并且没有其他线程可在初始化完成前对其进行处理，条件竞争终止于初始化阶段，这样比在之后再去处理好的多。在只需要一个全局实例情况下，局部static变量提供一个替代std::call_once的方案。
  17. 2保护读多写少的数据结构(读写锁优化)
    • C++17标准库提供了两种非常好的互斥量std::shared_mutex和std::shared_timed_mutex。C++14只提供了std::shared_timed_mutex，在C++11中并未提供这种类型的互斥量。如果你还在用支持C++14标准之前的编译器，那你可以使用Boost库中实现的互斥量。std::shared_mutex和std::shared_timed_mutex的不同点在于，std::shared_timed_mutex支持更多的操作方式，std::shared_mutex有更好的性能，因而不支持更多的操作。
    • 比起使用std::mutex，不如使用std::shared_mutex来做同步。对于更新操作，可以使用std::lock_guard 和 std::unique_lock 上锁。作为 std::mutex 的替代方案，与 std::mutex 所做的一样，这就能保证更新线程的独占访问。因为其他线程不需要去修改数据结构，所以其可以使用 std::shared_lock 获取访问权。这种RAII类型模板是在C++14中的新特性，这与使用std::unique_lock一样，除非多线程要在同时获取同一个std::shared_mutex上的共享锁。唯一的限制：当任一线程拥有一个共享锁时，这个线程就会尝试获取一个独占锁，直到其他线程放弃他们的锁；同样的，当任一线程拥有一个独占锁时，其他线程就无法获得共享锁或独占锁，直到第一个线程放弃其拥有的锁。
  18. 3嵌套锁
    • 当一个线程已经获取一个std::mutex时(已经上锁)，并对其再次上锁，这个操作就是错误的，并且继续尝试这样做的话，就会产生未定义行为。然而，在某些情况下，一个线程尝试获取同一个互斥量多次，而没有对其进行一次释放是可以的。之所以可以，是因为C++标准库提供了std::recursive_mutex类。
    • 大多数情况下，当需要嵌套锁时，就要对代码设计进行改动。嵌套锁一般用在可并发访问的类上，所以使用互斥量保护其成员数据。每个公共成员函数都会对互斥量上锁，然后完成对应的操作后再解锁互斥量。不过，有时成员函数会调用另一个成员函数，这种情况下，第二个成员函数也会试图锁住互斥量，这就会导致未定义行为的发生。“变通的”解决方案会将互斥量转为嵌套锁，第二个成员函数就能成功的进行上锁，并且函数能继续执行。
    • 但是，不推荐这样的使用方式，因为过于草率，并且不合理。特别是，当锁被持有时，对应类的不变量通常正在被修改。这意味着，当不变量正在改变的时候，第二个成员函数还需要继续执行。一个比较好的方式是，从中提取出一个函数作为类的私有成员，并且让其他成员函数都对其进行调用，这个私有成员函数不会对互斥量进行上锁(在调用前必须获得锁)。然后，你仔细考虑一下，在这种情况调用新函数时，数据的状态。

总结

  • 当多个线程间的共享数据发生恶性条件竞争时，如何使用std::mutex 和 避免死锁。C++标准库提供std::lock()防止死锁，互斥量并不是灵丹妙药，也还有自己的问题(比如：死锁)。其他用于避免死锁的技术，之后了解了锁的所有权转移，以及一些围绕如何选取适当粒度锁产生的问题。最后，在具体情况下讨论了其他数据保护的方案，例如: std::call_once() 和 std::shared_mutex。
  • 还有一个方面没有涉及到，那就是等待其他线程作为输入的情况。我们的线程安全栈，仅是在栈为空时，抛出一个异常，所以当一个线程要等待其他线程向栈压入一个值时(这是线程安全栈的主要用途之一)，它需要多次尝试去弹出一个值，当捕获抛出的异常时，再次进行尝试。这种消耗资源的检查，没有任何意义。并且，不断的检查会影响系统中其他线程的运行，这反而会妨碍程序的运行。我们需要一些方法让一个线程等待其他线程任务完成，但在等待过程中不占用CPU。第3章中，会建立一些保护共享数据的工具，还会介绍一些线程操作的同步机制；第5章中，如何构建基于锁的可复用的数据结构。
